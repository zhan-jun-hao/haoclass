经过联调已经能够下单支付 但是我冒烟的时候有个问题就是 如果订单支付失败了 优惠券还是核销了 

# 1.原因

根因是 **远程核销优惠券放在了本地履约事务中间**

`@Transactional` 只能管 **当前服务自己的数据库连接**，也就是 main-service 的库。它管不了 coupon-service 的数据库

# 2.解决

把本地事务和远程事务分开成两个事务

```java
  @Override
    public void handlePaymentSuccess(PaymentSuccessRequest request) {
        validatePaymentSuccessRequest(request);
        // 1.先处理本地事务
        CourseOrder courseOrder = transactionTemplate.execute(status ->
                handlePaymentSuccessInLocalTransaction(request));

        // 2.再处理优惠券事务
        if (courseOrder != null && courseOrder.getUserCouponId() != null) {
            useCoupon(courseOrder.getId(), courseOrder.getUserCouponId());
        }
    }
```

# 3.情况1

如果本地事务出现异常，不会核销优惠券，且本地事务回滚

# 4.情况2

如果本地事务成功，优惠券事务失败，那优惠券没有成功核销

但是可以接收，对用户的伤害比前者低，企业里经典的做法是

```bash
main-service 本地事务成功后
写一条 coupon_use_message 待核销消息
定时任务或 MQ 去调用 coupon-service
失败就重试
成功就标记完成
```

main-service 出问题会回滚，而且优惠券不会核销
如果 main-service 成功后 coupon-service 出问题，就靠幂等重试补偿

# 5.强事务

在openFeign远程调用的过程中，非常常见的情况就是遇到事务问题

强事务就是事务执行过程中，始终保证数据的一致性。只要事务没有成功，其他人就看不到中间状态；一旦事务失败，所有操作全部回滚。

比如用Seata来保证强事务

# 6.弱事务

即：保证最终一致性

它不是立刻一致，而是允许短时间内存在不一致，之后通过补偿机制恢复。

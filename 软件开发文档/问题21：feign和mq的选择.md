| 权衡点         | Feign                                                        | MQ                                                           |
| :------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **失败处理**   | 立即报错，调用方可以重试或熔断。但重试意味着用户还在等。     | **天然适合最终一致性**。扣款后发消息失败，可以用事务消息或本地消息表保证消息必达。 |
| **链路追踪**   | 很容易，一次调用就是一个 Trace。                             | 异步后追踪变复杂，需要手动传递 TraceId 到消息里，再在消费端解析串联。 |
| **系统耦合度** | **高**。A 需要知道 B 的接口定义（字段、URL），B 变更会直接影响 A。 | **低**。A 和 B 只依赖消息的结构。这也是你上一轮问到的“用 string 还是对象”的问题所在，用 JSON String 耦合度最低。 |
| **运维复杂度** | 低，就服务本身。                                             | 高，多了一个必须高可用的消息中间件，消息堆积、丢失都是问题。 |

如果需要立刻拿到数据，那就用feign，同步和异步的关系
